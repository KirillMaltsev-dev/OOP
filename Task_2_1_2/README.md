# Task_2_1_2

## Задача

В проекте реализована распределённая версия задачи поиска хотя бы одного составного числа в массиве целых чисел.

Алгоритм получает на вход массив чисел и возвращает:
- `true`, если в массиве найдено хотя бы одно непростое число
- `false`, если все числа простые

В отличие от задачи `2_1_1`, вычисление распределяется между несколькими вычислительными узлами одной подсети.  
В реализации не используются сторонние библиотеки и Java RMI.

## Общая идея решения

Система построена по схеме `координатор - вычислительные узлы`.

Есть два типа процессов:
- `Main` - координатор, который принимает входной массив, разбивает его на части и распределяет задачи между узлами
- `WorkerMain` - вычислительный узел, который принимает часть массива, проверяет числа на простоту и отправляет результат обратно

Координатор делит исходный массив на чанки фиксированного размера и формирует для каждого чанка отдельную задачу.  
Далее задачи распределяются между доступными воркерами по TCP-соединению.

## Архитектура

В проекте используются следующие основные классы:
- `DistributedPrimeChecker` - координатор распределённых вычислений
- `DistributedWorker` - серверная часть вычислительного узла
- `WorkerEndpoint` - описание адреса узла в формате `host:port`
- `ChunkTask` - задача на проверку части массива
- `Protocol` - константы и контрольная сумма для обмена сообщениями
- `PrimeChecker` - базовая логика проверки чисел на простоту
- `Main` - точка входа координатора
- `WorkerMain` - точка входа воркера

## Топология сети

Используется простая звездообразная топология:
- один координатор
- один или несколько воркеров
- все воркеры доступны координатору по TCP в рамках одной подсети

Координатор знает список узлов заранее через аргументы командной строки:
`host1:port1,host2:port2,...`

## Межсетевое взаимодействие

Обмен между координатором и воркером происходит по TCP-сокетам.

Для каждой задачи координатор:
1. устанавливает соединение с воркером
2. отправляет тип запроса
3. отправляет идентификатор задачи
4. отправляет длину чанка
5. отправляет все числа из чанка
6. отправляет контрольную сумму

Воркер:
1. принимает запрос
2. проверяет корректность данных
3. обрабатывает полученный чанк
4. возвращает результат

Типы сообщений:
- `CHECK_TASK` - запрос на обработку чанка
- `RESULT` - успешный ответ
- `ERROR` - сообщение об ошибке

## Сохраняемость и целостность данных

Для контроля корректности передачи используется контрольная сумма `CRC32`.

Перед отправкой чанка координатор вычисляет checksum по массиву чисел и передаёт её вместе с данными.  
После получения воркер пересчитывает checksum и сравнивает её с присланной.

Если контрольные суммы не совпадают, воркер возвращает `ERROR`, а задача считается неуспешной.

Таким образом обеспечивается базовая защита от повреждения передаваемых данных.

## Отказоустойчивость

В системе предусмотрена базовая отказоустойчивость.

Если вычислительный узел:
- недоступен
- не отвечает
- разрывает соединение
- возвращает ошибку

то координатор не считает задачу потерянной.  
Чанк возвращается обратно в очередь и может быть повторно назначен другому доступному воркеру.

Если найдено составное число, координатор завершает вычисление досрочно и возвращает `true`.

Если все узлы недоступны и обработать оставшиеся чанки невозможно, координатор завершает работу с ошибкой.

## Идентичность и состояние распределённых объектов

Каждый чанк имеет уникальный идентификатор задачи `taskId`.

Идентификатор используется для:
- сопоставления запроса и ответа
- контроля корректности ответа от узла
- логического разделения задач внутри одного вычисления

Состояние вычисления хранится на стороне координатора:
- очередь необработанных чанков
- количество оставшихся задач
- факт нахождения составного числа
- информация об ошибках выполнения

Воркеры являются без состояния между запросами:
- они получают чанк
- вычисляют результат
- отправляют ответ
- не хранят историю предыдущих задач

## Алгоритм завершения работы

В распределённой версии вычисление завершается по одному из двух условий:
- найдено хотя бы одно составное число
- обработаны все чанки и составных чисел не найдено

Координатор использует досрочное завершение:
если любой воркер сообщает о найденном составном числе, остальные вычисления больше не нужны.
