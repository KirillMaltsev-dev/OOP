# Task_2_1_1

## Задача

В проекте реализованы 3 алгоритма поиска хотя бы одного составного числа в массиве целых чисел.

Алгоритм получает на вход массив чисел и возвращает:
- `true`, если в массиве найдено хотя бы одно непростое число
- `false`, если все числа простые

Реализованы следующие варианты решения:
- `seq` - последовательный алгоритм в 1 поток
- `thread` - параллельный алгоритм с использованием `java.lang.Thread`
- `stream` - параллельный алгоритм с использованием `parallelStream`

## Реализация

Проект содержит три реализации проверки:
- `SequentialPrimeChecker` - последовательный поиск
- `ParallelThreadPrimeChecker` - параллельный поиск с разбиением массива на части между потоками
- `ParallelStreamPrimeChecker` - параллельный поиск через `parallelStream`

Проверка числа на простоту выполняется перебором делителей до `sqrt(n)`.

## Сценарии исследования производительности

Для сравнения производительности были подготовлены 4 набора входных данных:

1. `small_small` - массив длины 300, состоящий из небольших простых чисел
2. `large_small` - массив длины 5000, состоящий из небольших простых чисел
3. `small_large` - массив длины 300, состоящий из больших простых чисел
4. `large_large` - массив длины 1500, состоящий из больших простых чисел

Такое разбиение позволяет отдельно оценить влияние:
- размера массива
- сложности проверки одного числа на простоту

## Конфигурации запуска

Для многопоточной версии с `Thread` использовались фиксированные конфигурации:
- 1 поток
- 2 потока
- 4 потока
- 8 потоков

Для версии `parallelStream` использовалась отдельная параллельная конфигурация, которая сравнивалась с остальными алгоритмами как самостоятельный способ распараллеливания.

## Методика измерений

Для каждого сценария выполнялось:
- 2 прогревочных запуска
- 5 измерений времени выполнения

По результатам измерений вычислялись:
- среднее время выполнения `avg`
- минимальное время
- максимальное время
- 90% доверительный интервал
- коэффициент ускорения относительно последовательной версии

Сырые результаты измерений сохраняются в папку `benchmark_results`.
Итоговые диаграммы сохраняются в папку `benchmark_diagram`.

## Вывод

На маленьких и лёгких входных данных последовательная версия часто оказывается быстрее или сопоставима с параллельными версиями, так как накладные расходы на создание потоков и синхронизацию оказываются заметными по сравнению с объёмом полезной работы.

При увеличении размера массива и сложности проверки одного числа эффективность параллельных алгоритмов становится выше. Наиболее заметный выигрыш ожидается на наборе `large_large`, где одновременно велики и размер входных данных, и стоимость проверки каждого числа.

Таким образом, распараллеливание особенно полезно на вычислительно тяжёлых сценариях, а на маленьких входах его преимущества могут не окупаться.
